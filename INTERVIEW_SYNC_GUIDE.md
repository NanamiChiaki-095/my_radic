# my_radic 分布式同步与一致性技术指南

## 1. 核心挑战：双写一致性 (The Dual-Write Problem)
**问题描述**：在搜索引擎中，我们需要同时更新数据库（MySQL）和索引（Indexer）。如果先写库成功，但索引更新失败，会导致“数据丢了”；如果先写索引，库失败，会导致“搜到不存在的数据”。

**我的方案**：**Transactional Outbox (事务发件箱模式)**
*   **实现逻辑**：
    1.  开启 MySQL 事务。
    2.  写入业务数据表 (`articles`)。
    3.  在**同一个事务**中，写入消息记录表 (`outbox`)。
    4.  提交事务。
*   **亮点**：利用本地数据库事务的 **ACID** 特性，保证了“业务数据”与“同步指令”的强一致性。只要文章发布成功，同步消息一定在数据库里。

---

## 2. 同步链路：准实时 (Near Real-Time) 管道
数据从 MySQL 到达搜索结果的过程：
1.  **Relay Service (中继器)**：后台协程轮询 `outbox` 表，将 `Pending` 状态的消息投递至 Kafka。
2.  **Kafka (解耦与削峰)**：作为消息骨干，解耦了高并发的写入请求与后端的索引构建。
3.  **Consumer Group (分片消费)**：
    *   Index Service 节点通过消费者组订阅 Kafka。
    *   **实现原理**：Kafka 自动将 Partition 分配给不同的索引节点，实现了**数据分片 (Sharding)**。
    *   **扩展性**：当数据量增大时，只需增加 Kafka 分区和索引节点，系统即可实现水平扩容。

## 3. 健壮性保障：幂等性与故障恢复
分布式环境下，网络抖动和崩溃是常态，我的系统做了如下设计：

### 3.1 幂等性 (Idempotency) —— 解决“重复同步”
*   **挑战**：Relay 或 Kafka 可能因为重试导致同一条消息发送多次。
*   **解决**：Indexer 的底层 `AddDocument` 逻辑天然幂等。`kvdb.Set(docId, bytes)` 是覆盖操作，`SkipList.Add` 会更新相同 ID 的分数值。无论收到几次同步指令，最终结果一致。

### 3.2 WAL 与状态恢复 —— 解决“节点崩溃”
*   **挑战**：倒排索引在内存中，节点重启会丢失所有索引。
*   **解决**：我将 **BoltDB (B+ Tree)** 作为本地 WAL (Write-Ahead Log)。同步消息进入节点后，先落磁盘 BoltDB（持久化），再进内存。
*   **收益**：重启时，节点直接扫描本地磁盘重建内存索引，无需回查 MySQL，恢复速度从“分钟级”提升到“秒级”。

## 4. 面试高频 Q&A (模拟对线)

**Q1: 为什么不直接在 API 逻辑里同步调用索引服务？**
*   **答**：1. **性能**：RPC 是同步阻塞的，会拖慢响应速度。 2. **可用性**：如果索引服务宕机，会导致整个发贴功能不可用。 3. **一致性**：无法保证 RPC 成功和数据库事务提交的原子性。

**Q2: 你的 Relay 轮询数据库会产生 IO 压力，有优化方案吗？**
*   **答**：目前的轮询适合中等规模场景。**进阶方案**是使用 **CDC (Change Data Capture)**。通过监听 MySQL 的 **Binlog** (例如使用 Canal 或 Debezium)，实现实时触发。这样可以消除 Relay 的查询压力，将同步延迟降低到毫秒级。

**Q3: 如果 Kafka 消息堆积了，你怎么排查和处理？**
*   **答**：
    1.  **排查**：通过监控查看 **Consumer Lag** (消费积压量)。
    2.  **分析**：查看 Index Service 的 CPU/磁盘 IO。通常瓶颈在 BoltDB 的写压力。
    3.  **处理**：
        *   **纵向**：增加单次批量消费的大小 (`BatchSize`)。
        *   **横向**：增加 Kafka 分区数，并启动更多的 Index Service 节点。
