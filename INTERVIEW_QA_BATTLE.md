# ⚔️ MyRadic 面试极客对线指南 (Battle Guide)

> **文档用途**：本指南模拟了**高级技术面试官**的深挖逻辑。请不要只背答案，而要理解其背后的**Trade-off (权衡)**。
> **核心原则**：面试官问 A，你回答 A，这叫及格；面试官问 A，你回答 A 的原理、A 的缺陷以及为什么不用 B，这叫**Offer**。

## 第一回合：数据结构与算法内核 (The Kernel)

### ❓ Q1: 你的倒排索引为什么用跳表 (SkipList) 而不用数组或红黑树？
**面试官心理**：想看你是不是只会调包，懂不懂底层数据结构的适用场景。

**🙋‍♂️ 你的回击**：
*   **针对数组**：数组虽然内存紧凑，但在**插入**文档时需要移动大量数据（O(N)），这在实时索引（Real-time Indexing）场景下是不可接受的。跳表插入是 O(log N)。
*   **针对红黑树**：
    1.  **区间查找**：倒排索引经常需要“查找 DocID > 10000 的所有文档”，跳表只需要找到节点然后向后遍历链表即可，而红黑树需要复杂的中序遍历回溯。
    2.  **并发优势**：红黑树的再平衡（Rebalance）涉及旋转，会锁住大片树结构；而跳表只需要局部 CAS 更新指针，天然适合高并发实现（虽然本项目目前是分段锁，但保留了无锁化的潜力）。
    3.  **多路归并优化**：最重要的是，跳表支持高效的 **`SkipTo`** 操作。在求 `A AND B` 交集时，如果 A 的当前 ID 是 1，B 是 1000，跳表可以直接从高层索引“跳”到 1000 附近，而红黑树很难做到这一点。

### ❓ Q2: 既然跳表这么好，为什么 MySQL 还是用 B+ 树？
**面试官心理**：看你是否理解“内存”与“磁盘”的差异。

**🙋‍♂️ 你的回击**：
*   **IO 为王**：跳表是基于指针的，节点分散在内存各处，缓存局部性（Cache Locality）较差，且指针层级高，会导致大量的**随机磁盘 IO**。
*   **B+ 树优势**：B+ 树的节点是一个 Page（如 16KB），一次 IO 能读入成百上千个 Key，极其适合磁盘存储。
*   **我的架构**：所以我采用了**存算分离**。
    *   **内存中**：用跳表（计算快，灵活性高）。
    *   **磁盘上**：用 BoltDB（底层是 B+ 树），利用其顺序写和 Page 缓存特性做持久化。这是取长补短的设计。

## 第二回合：分布式一致性 (Consistency)

### ❓ Q3: 你用了 Transactional Outbox，但如果 Relay 服务挂了怎么办？消息不就发不出去了？
**面试官心理**：考察你对高可用的理解。

**🙋‍♂️ 你的回击**：
*   **不会丢数据**：Relay 挂了没事，因为消息已经**持久化在 MySQL** 的 `outbox` 表里了。
*   **恢复机制**：Relay 重启后（或者 K8s 拉起新的 Pod），会重新扫描 `status=Pending` 的记录。
*   **延迟问题**：这确实会导致同步延迟增加。
*   **进阶优化 (加分项)**：为了避免 Relay 单点故障，可以让 Relay 也是分布式的（利用 Redis 分布式锁抢占任务，或者基于 ID 取模分片扫描），或者直接上 **CDC (Canal/Debezium)** 监听 Binlog，完全去掉轮询的 Relay 服务。

### ❓ Q4: 如果 Relay 以为发送失败了（网络超时），但实际上 Kafka 收到了，重试发送会导致重复消费吗？
**面试官心理**：考察**幂等性 (Idempotency)**。

**🙋‍♂️ 你的回击**：
*   **一定会重复**：At-Least-Once 语义下，重复是必然的。
*   **我不在乎**：我的 Consumer 端实现了**全链路幂等**。
    *   **KV 存储**：`boltDB.Put(key, val)` 是覆盖写。
    *   **倒排索引**：`SkipList.Add(term, docId)` 内部会检查 DocID 是否已存在。如果存在，我会更新其 Score/Feature，而不会插入两个相同的 DocID。
    *   **结论**：无论这条消息发来多少次，系统的最终状态都是一致的。

## 第三回合：高并发与 Go 语言特性 (Concurrency)

### ❓ Q5: 你的 SingleFlight 是怎么实现的？如果那个“唯一的请求”一直卡住不返回怎么办？
**面试官心理**：考察源码细节和故障处理。

**🙋‍♂️ 你的回击**：
*   **原理**：内部维护一个全局 `map[string]*call` 和 `sync.WaitGroup`。
*   **卡死风险**：这确实是个坑。如果第一个请求死锁或网络挂起，后续 1000 个等待的请求都会陪葬（Goroutine 泄漏）。
*   **我的解法**：
    1.  **Context 超时**：在调用 gRPC 时必须传入带有 `Timeout` 的 Context。
    2.  **DoChan**：SingleFlight 提供了 `DoChan` 方法，支持 select 异步等待。
    3.  **Panic 保护**：SingleFlight 内部处理了 Panic，确保不会因为一个请求崩溃带崩全站。

### ❓ Q6: 在压测中，你的 QPS 达到了 4000+，当时的瓶颈在哪里？CPU 还是网络？
**面试官心理**：看你有没有真的做过压测，还是瞎编的数据。

**🙋‍♂️ 你的回击**：
*   **现象**：在 4000 QPS 时，CPU 使用率飙升。
*   **瓶颈分析**：
    1.  **GC 压力**：大量的搜索请求产生了海量的临时对象（SkipList 节点遍历、结果集 Slice 扩容、Protobuf 序列化）。Go 的 GC 标记阶段消耗了大量 CPU。
    2.  **JSON 序列化**：Gin 返回大 JSON 结果是非常消耗 CPU 的。
*   **优化思路**：
    *   使用 `sync.Pool` 复用 `SearchResponse` 对象，减少 GC。
    *   使用 `Sonic` 或 `EasyJSON` 替换标准库的 JSON 库。

## 第四回合：系统设计与扩展性 (System Design)

### ❓ Q7: 你现在是单机索引，如果我有 10 亿文档，单机存不下怎么办？
**面试官心理**：考察**分库分表/Sharding** 的设计能力。

**🙋‍♂️ 你的回击**：
*   **垂直拆分**：这已经是微服务了，Index Service 和 API Gateway 已经分离。
*   **水平拆分 (Sharding)**：
    *   **目前架构**：我已经引入了 Kafka Consumer Group。
    *   **方案**：增加 Kafka Partition 数量（比如 100 个），然后启动 100 个 Index Service 节点。
    *   **路由策略**：
        1.  **写**：文档 ID Hash 取模，决定发到哪个 Kafka Partition。
        2.  **读**：Gateway 需要通过 gRPC **广播 (Scatter-Gather)** 给所有 Index 节点，然后将结果**归并排序**。
*   **挑战**：Scatter-Gather 会导致“长尾效应”（最慢的节点决定整体延迟）。需要引入**只有 P95 节点返回即可**的降级策略。

### ❓ Q8: 如果我要做“实时热搜排行榜”，你的架构怎么改？
**面试官心理**：考察对 Redis 场景的运用。

**🙋‍♂️ 你的回击**：
*   **非侵入式修改**：在 API Gateway 的 `/search` 接口中埋点。
*   **异步处理**：不要阻塞搜索主流程。把搜索词丢进一个 Go Channel 或发给 Kafka。
*   **流式计算**：后台起一个 Worker，消费搜索词。
*   **存储**：使用 Redis 的 **ZSET (Sorted Set)**。
    *   `ZINCRBY daily_hot_search <keyword> 1`
    *   定界：每 10 分钟或 1 小时新建一个 Key，做滑动窗口统计。
