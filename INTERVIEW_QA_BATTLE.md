# MyRadic 面试问答手册（场景化详解版）

> 适用目标：你在面试里“讲清楚为什么这样设计”，而不是只给结论。  
> 建议表达顺序：**场景 -> 问题 -> 方案 -> 为什么是它 -> 代价与边界 -> 下一步**。

---

## 0. 开场方法（先把节奏拿回来）

当面试官问技术问题时，你可以先用这一句起手：

“我先结合一个真实场景说一下：当时我们遇到的问题是 X，所以选了 Y 方案，Y 的收益是 A，代价是 B，我也准备了后续优化 C。”

这样你会显得“有工程全局观”，不是背八股。

## 1) 项目总览类（先讲全局）

## Q1：你这个项目核心解决了什么问题？

**场景**
- 一个内容平台需要支持“发文后可搜索”，并且高并发下搜索不能抖动太厉害。

**问题**
- 写入和索引更新容易不一致。
- 热点搜索会打爆下游。
- 出问题时很难定位瓶颈。

**回答模板**
- 我主要解决三件事：  
  1) 用 Transactional Outbox + Kafka 做写入到索引的最终一致性；  
  2) 用 Redis + SingleFlight + 分片并发提升查询吞吐；  
  3) 用 Prometheus + pprof + 压测闭环做可观测和性能定位。  

**为什么这样说**
- 这三个点覆盖了面试官最关心的：正确性、性能、可运维性。

---

## Q2：最体现你工程能力的改动是什么？

**场景**
- 压测时吞吐上不去，且尾延迟高。

**问题**
- 不知道是业务逻辑慢，还是平台/日志开销导致。

**回答模板**
- 我不是直接改代码，而是先 pprof 定位热点。  
- 定位到日志写路径占比高后，做了结构化日志、异步写队列、成功请求采样。  
- 再跑同口径压测验证变化，形成“观测 -> 改造 -> 回归”的闭环。

**代价与边界**
- 日志采样会损失部分细粒度信息，所以保留了错误日志全量输出。

## 2) 一致性与可靠性（高频深挖）

## Q3：为什么不用“写 DB 成功后，同步 RPC 更新索引”？

**场景**
- 用户发布文章，接口返回成功后希望可搜。

**问题**
- 同步 RPC 模式下，DB 与索引更新不是一个原子操作：
  - DB 成功 + RPC 失败：发文成功但搜不到；
  - RPC 成功 + DB 回滚：索引出现脏数据。

**方案**
- Transactional Outbox：业务数据和 outbox 消息同事务写入 MySQL；异步投递到 Kafka；Indexer 消费构建索引。

**为什么选它**
- 它把“消息不会丢”这个问题交给数据库事务解决，工程上稳妥且易落地。

**代价与边界**
- 不是强实时，一致性窗口取决于 relay + Kafka + 消费速度。

**追问准备**
- 如果要更低延迟，可评估 CDC（Debezium）替代轮询 relay。

---

## Q4：Relay 挂了会丢消息吗？

**场景**
- 投递进程崩溃或发布期间短暂不可用。

**回答模板**
- 不会丢。因为消息先落在 `outbox` 表。  
- Relay 恢复后继续扫描 `pending` 补发。  
- 影响是“可搜延迟变大”，不是“数据丢失”。

**为什么这样答**
- 你明确区分了“正确性”与“时效性”，这是关键。

---

## Q5：消息重复消费怎么办？

**场景**
- Kafka/网络重试导致同一消息被消费多次。

**方案**
- 设计接受 `at-least-once`，消费端做幂等。
- 以 `doc_id` 为主键，写入覆盖语义，不做重复累加。

**为什么选它**
- 比追求 exactly-once 成本低很多，且在搜索场景足够实用。

**代价**
- 需要严格约束更新逻辑，确保重复消费不会改变最终状态。

---

## Q6：你怎么证明一致性方案有效？

**场景化验证思路**
1. 人为让 relay 在“发 Kafka 成功、更新 outbox 状态前”崩溃。  
2. 重启后观察是否重发。  
3. 验证索引最终状态是否与单次消费一致。  

**回答要点**
- 我做的是“故障推演 + 幂等验证”，不是只看 happy path。

## 3) 搜索性能与架构取舍（高频）

## Q7：为什么倒排结构用 SkipList？

**场景**
- 数据持续写入，同时要支持布尔查询（AND/OR）和较低延迟。

**为什么不用简单数组**
- 数组在频繁插入和合并时维护成本高，且复杂查询时跳跃能力差。

**为什么用 SkipList**
- 插入/查找复杂度更平衡；
- 支持 `SkipTo`，在交并集计算时能减少无效遍历；
- 实现复杂度可控，便于迭代。

**边界**
- 内存结构对内存占用更敏感，需要配套持久化与恢复策略。

---

## Q8：SingleFlight 具体价值是什么？

**场景**
- 热门词同时涌入（如“golang”）时，缓存刚好 miss。

**问题**
- 没有归并会导致大量并发回源，放大下游压力。

**方案**
- 对同 key 查询做请求合并，一次回源，多请求共享结果。

**结果**
- 显著降低击穿期间的下游 QPS 峰值。

**代价**
- 单个“首请求”慢会拖住同组请求，必须配合超时控制。

---

## Q9：你做过哪些“有证据”的性能优化？

**场景**
- 火焰图显示 `runtime.cgocall/syscall write` 占比高。

**方案**
- 把日志改为结构化 + 异步 + 采样（成功请求降噪，错误日志保留全量）。

**为什么是这个优先级**
- 日志属于高频公共路径，优化收益面广、风险相对可控。

**验证方式**
- 同口径压测对比：吞吐、错误率、p95/p99、CPU 热点变化。

---

## Q10：你的压测上限和瓶颈判断是什么？

**场景**
- 在 Docker Desktop + kind（3 worker 逻辑集群）压测。

**观察**
- 稳态可到约 `9k RPS`；平台状态好时冲到 `13k~14k`，但尾延迟明显恶化。
- 高压下出现 K8s API / Docker TLS handshake timeout。

**判断**
- 这是“平台层先到边界”（运行时、网络、控制面），不是纯业务函数瓶颈。

**为什么这结论可靠**
- 有压测数据 + 平台报错 + pprof 共同印证，而不是单指标猜测。

## 4) Go 并发与工程细节（容易追问）

## Q11：如何避免 map 并发问题？

**场景**
- 网关动态更新 indexer 节点连接表，同时有高并发查询读取路由。

**方案**
- `map + RWMutex` 管理共享状态；
- 慢操作（例如 `Dial`）放在锁外；
- 锁内只做“提交变更”。

**为什么不用 sync.Map**
- 这里需要复合操作和批量一致性控制，`RWMutex` 更清晰可控。

---

## Q12：WebSocket 在 shutdown 怎么处理？

**场景**
- 进程退出时，如果不主动关闭 WS，可能出现连接泄漏和客户端长时间挂起。

**方案**
- 用 `ws_hub` 统一管理连接：`Add/Remove/CloseAll`。
- shutdown 时先停止接收新请求，再按 deadline 主动关闭存量连接。

**收益**
- 退出行为可预期，减少灰度/滚动发布期的异常连接。

---

## 5) 分布式与 K8s（中高级）

## Q13：分片是怎么做的？

**场景**
- Index 服务要横向扩展。

**方案**
- `StatefulSet` 副本与 shard 绑定（通过 `HOSTNAME` 推导 shard id）。
- Gateway 根据 shard 路由并发查询，最后聚合结果。

**边界**
- 聚合路径会引入“慢分片拖尾效应”，需要关注长尾治理。

---

## Q14：为什么 3 worker 没明显超过单机？

**场景**
- kind 三 worker 都跑在同一宿主机。

**本质**
- 这是逻辑分布式，不是物理扩容；算力池没变，平台开销反而增加。

**回答模板**
- “这个结果符合预期。要验证分布式收益，需要多物理机并且压测机与服务机隔离。”

## 6) 反思与下一步（加分）

## Q15：你认为当前最大短板是什么？

**建议回答**
- Relay 轮询在高写入下对主库有压力；
- 平台层极限压测稳定性不足，影响高压结论可重复性。

## Q16：下一步优先级？

**建议回答**
1. 固定回归基线（3k/6k/9k/12k）  
2. 压测联动采集 pprof + Prometheus + 节点资源  
3. 评估 CDC 替代 relay 轮询  
4. 多机环境验证真实分布式收益

## 8) 30 秒与 3 分钟版本（可直接练）

## 10) 工程问题复盘案例库（可直接放进项目经历）

> 用法：从下面挑 3~5 个最贴近你真实改动的案例讲，不要一次全讲。  
> 结构建议：**现象 -> 根因 -> 方案 -> 结果 -> 复盘**。

### 案例 1：网关重载节点时偶发高延迟尖刺

- **现象**：服务节点变更后，搜索接口偶发 p99 尖刺。  
- **根因**：节点重载逻辑里把 `grpc.Dial` 放在写锁内，慢拨号阻塞了读路径。  
- **方案**：改成“锁外拨号、锁内提交”（两阶段更新连接表）。  
- **结果**：节点变更期间尾延迟明显收敛。  
- **复盘**：慢操作尽量不要放在共享写锁里。

### 案例 2：停机时 WebSocket 连接泄漏

- **现象**：服务退出后客户端仍有残留连接，重启期间行为不稳定。  
- **根因**：缺少统一连接生命周期管理，shutdown 时未主动回收存量连接。  
- **方案**：引入 `ws_hub`，统一 `Add/Remove/CloseAll`，停机阶段按 deadline 主动关闭。  
- **结果**：滚动重启更平滑，连接残留显著减少。  
- **复盘**：长连接场景必须有显式的“连接回收策略”。

### 案例 3：高并发压测时吞吐上不去

- **现象**：QPS 提升后 CPU 占用很高，但业务函数占比不明显。  
- **根因**：pprof 显示大量时间耗在日志写路径（`syscall/cgocall`）。  
- **方案**：日志结构化 + 异步写队列 + 成功请求采样降噪。  
- **结果**：日志开销下降，压测稳定性提升。  
- **复盘**：先定位再优化，避免“拍脑袋改代码”。

### 案例 4：缓存击穿导致下游压力暴涨

- **现象**：热点词缓存失效瞬间，下游 indexer QPS 暴涨。  
- **根因**：同一个 query 同时回源，形成并发放大。  
- **方案**：在 gateway 增加 SingleFlight，同 key 请求合并回源。  
- **结果**：击穿窗口内下游峰值明显降低。  
- **复盘**：缓存之外，还要处理“回源风暴”。

### 案例 5：重复消息导致索引重复写风险

- **现象**：Relay/Kafka 重试后同一消息可能多次到达消费端。  
- **根因**：链路采用 `at-least-once`，重复消费是正常语义。  
- **方案**：以 `doc_id` 作为幂等键，消费侧采用覆盖写与去重更新。  
- **结果**：重复投递不影响最终索引状态。  
- **复盘**：重复消息不是异常，要在设计里“正面接住”。

### 案例 6：网关启动时报 “no available index_service instances”

- **现象**：gateway 启动失败，提示没有可用 index_service 节点。  
- **根因**：依赖启动顺序问题，indexer 尚未注册到 etcd。  
- **方案**：先确保 indexer 就绪并注册，再启动 gateway；同时完善启动期重试与 readiness 判定。  
- **结果**：启动流程可预测，不再“碰运气”。  
- **复盘**：分布式系统要把“依赖就绪”视为一等公民。

### 案例 7：K8s 部署后 Pod 启动失败（配置挂载问题）

- **现象**：gateway Pod 报 `configmap not found` 或反复重启。  
- **根因**：资源 apply 顺序不稳定，配置资源未先创建。  
- **方案**：固定部署顺序（基础设施 -> 配置 -> 应用）并加 rollout 检查。  
- **结果**：部署成功率提升，问题可快速定位。  
- **复盘**：部署流程要“可重复”，而不是“手动记忆”。

### 案例 8：分布式压测 12k+ 后集群不稳定

- **现象**：高压下出现 `TLS handshake timeout`、Docker API 500。  
- **根因**：kind 多 worker 都在同一宿主机，平台层先达到资源边界。  
- **方案**：分档压测（3k/6k/9k/12k）+ 稳态复测，分离“业务瓶颈”与“平台瓶颈”。  
- **结果**：压测结论更可信，知道该优化代码还是扩容环境。  
- **复盘**：压测要先保证环境可重复，再谈极限数值。

### 案例 9：服务下线期间仍接入新流量

- **现象**：发布/重启时出现少量 5xx。  
- **根因**：优雅停机与 readiness 协同不足，下线窗口仍有请求进入。  
- **方案**：停机时先标记 not-ready，再执行 graceful shutdown，给在途请求缓冲时间。  
- **结果**：发布期错误率下降。  
- **复盘**：优雅停机是“流量治理问题”，不只是 `server.Shutdown()`。

### 案例 10：压测结论前后不一致

- **现象**：同参数压测结果波动大，无法评估改动效果。  
- **根因**：测试口径不一致（缓存状态、压测时长、并发配置不同）。  
- **方案**：固定测试模板与输出格式（warm/cold、固定档位、统一 CSV）。  
- **结果**：可对比、可回归、可解释。  
- **复盘**：性能优化最怕“不可复现”，先统一方法论。

### 面试表达建议（可套用）

“这个问题我当时不是直接改代码，而是先把现象量化，然后用日志和 pprof定位根因，接着做了一个最小可验证改动，最后用同口径压测回归。这样可以确保优化是有效且可复现的。”
