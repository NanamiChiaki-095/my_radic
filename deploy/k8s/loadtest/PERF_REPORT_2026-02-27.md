# 分布式压测报告（2026-02-27）

> 测试日期：2026-02-27  
> 测试环境：Windows + Docker Desktop + kind（1 control-plane + 3 worker）  
> 结论先行：当前拓扑下可稳定跑到 `9k RPS`，`12k` 以上开始出现明显平台侧不稳定信号。

---

## 1. 测试目标

验证三 worker 集群下的搜索吞吐上限，确认：

- 是否显著高于此前单机链路数据
- 高压时是业务瓶颈还是平台瓶颈

---

### 2.2 分档

## 5. 瓶颈定位结论

### 5.1 不是“纯代码瓶颈”

如果是纯应用热点，通常会看到：

- 压测稳定跑完（只是延迟上升）
- `kubectl` 与 Docker API 仍稳定

本次实际出现的是平台链路异常（K8s API / Docker daemon 超时），说明先碰到宿主机与容器平台上限。

### 5.2 关键原因

1. kind 三 worker 都在同一台物理机，算力池没有真正扩容。  
2. 高并发下，容器网络 + 运行时 + 控制面开销放大。  
3. Docker Desktop / WSL2 在极限并发下更容易先失稳。  

---

## 6. 对当前项目的实际建议

### P0：先把测试基线稳定住

- 固定回归档位：`3000/6000/9000/12000`，每档 60 秒。
- 不在同轮内直接冲 `15000+`，先保证可重复性。

### P1：把“业务开销”和“平台开销”拆开看

- 同步采集：
  - Gateway / Indexer 的 pprof（CPU、mutex、block）
  - Prometheus 中请求时延、错误率、goroutine、GC
  - 节点 CPU / 内存 / 网络利用率

### P2：验证真正分布式收益

- 使用多台物理机或云主机（不是单机 kind）
- 压测机与服务机分离
- 再做同档位压测对比

---

## 7. 下一轮复测标准（建议）

复测通过标准：

- `6000` 档：`fail_rate=0`，`p95 < 20ms`
- `9000` 档：`fail_rate=0`，`p95 < 50ms`
- `12000` 档：`fail_rate < 1%`，且平台不出现 API/TLS 大量超时

若不满足，先修平台稳定性，再继续冲高吞吐。
